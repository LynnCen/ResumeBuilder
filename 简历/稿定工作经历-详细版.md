#### 我的核心工作

##### 1. Monorepo架构设计 🔥

**四层分层架构：**
- **应用层（apps/）**：路由、配置、组装、部署（gaoding、insmind、open-platform等8+应用）
- **领域层（domains/）**：特定业务领域完整逻辑（editor、commerce等垂直切分）
- **共享层（packages/）**：跨领域通用代码（ui组件、utils工具、hooks等）
- **平台层（platform/）**：跨切面技术能力（config、monitoring、auth、storage）
- 实现单向依赖约束（上层可依赖下层，下层不能依赖上层），通过tags标签和boundaries规则强制架构边界

**Catalog依赖管理：** 🔥 技术创新点
- 实现Catalog依赖目录，统一管理45+核心依赖版本（Vue、React、TypeScript、Vite、PixiJS等）
- 避免版本碎片化问题，支持多版本catalog（Vue2/Vue3技术栈共存）
- **开发自动化Catalog更新机器人**：
  - 定时检测依赖更新（npm registry）
  - 自动创建MR（包含依赖收集、版本分发、格式化、MR创建全流程）
  - 减少人工维护成本90%

**workspace协议管理：**
- 设计workspace协议管理内部依赖（workspace:*、workspace:^、workspace:~）
- 支持本地链接（开发时）与发布时自动转换为语义化版本
- 配置Overrides强制版本覆盖，解决第三方包安全漏洞问题

##### 2. Turbo构建性能优化 🔥

**增量构建机制：**
- 设计任务依赖图（build → test → deliverable → deploy → verify）
- 配置dependsOn实现拓扑排序确保构建顺序（^build表示上游依赖的build任务）
- 识别可并行的任务（无依赖关系），最大化CPU利用率

**智能缓存策略：**
- **CacheKey计算公式**：Hash(任务配置 + inputs + env + upstreamOutputs)
- **精确定义inputs**：排除不影响构建的文件（!**/*.test.*、!**/*.md）
  - 避免README修改导致缓存失效
  - 提升缓存命中率至85%+
- **远程缓存**：接入Turbo远程缓存（TURBO_TOKEN），团队共享构建结果
- **环境变量策略**：globalEnv影响缓存、globalPassThroughEnv不影响缓存

**性能优化效果：**
- CI/CD Pipeline耗时从30分钟优化至12分钟
- 增量场景优化至3-5分钟
- 缓存命中率85%+
- 开发反馈周期从30分钟缩短至5分钟

**构建性能分析：**
- 开发turbo-analyzer包装脚本，自动生成构建性能分析报告（.turbo/runs/）
- 持续监控缓存命中率、任务执行时间等关键指标

##### 3. CI/CD Pipeline架构设计 🔥

**7个Stage流程设计：**
1. **Bot Stage**：自动化维护任务（Catalog更新、包索引生成、编辑器代码同步）
2. **Check Stage**：代码检查（ESLint、Prettier、TypeScript）
3. **Test Stage**：自动化测试（单元测试、集成测试）
4. **Changeset Stage**：版本管理（收集changesets、创建版本MR）
5. **Deliverable Stage**：制品生成（npm包、Docker镜像）
6. **Deploy Stage**：环境部署（Review/Staging/Production）
7. **Verify Stage**：部署验证（健康检查、烟雾测试）

**GitLab MR Bot自动化：**
- 开发GitLab MR Bot，统一自动化MR工作流
- 实现syncAndCreateMR通用接口，支持：
  - Catalog更新（依赖版本统一管理）
  - 包索引生成（自动维护包列表）
  - 编辑器代码同步（跨仓库代码同步）
- 所有维护任务自动化，减少人工干预90%

**分支冻结机制：** 🔥 技术创新点
- **问题**：版本MR等待Pipeline期间，其他代码合并导致无限循环
- **创新解决方案**：通过GitLab API动态修改分支保护规则
  - freezeBranch()：设置access_level为0，禁止所有人推送和合并
  - 等待Pipeline成功后临时恢复权限自动合并
  - unfreezeBranch()：完成后解冻恢复原始规则
  - 使用resource_group机制确保全局同时只运行一个实例，避免并发冲突
- **价值**：彻底解决版本MR无限循环问题，版本发布人工干预减少90%

**多环境部署策略：**
- 实现Review、Staging、Production三种环境
- **Review环境**：基于MR号动态创建（review/mr123），MR关闭后自动销毁
- **环境隔离**：CI_ENVIRONMENT_TYPE + CI_ENVIRONMENT_INSTANCE动态计算
- **Docker容器化**：多阶段构建优化镜像体积，pnpm deploy生成.pruned精简依赖树
- 集成K8s容器编排，支持灰度发布、蓝绿部署、金丝雀发布

##### 4. Changesets版本管理自动化

**完整工作流：**
1. **开发阶段**：开发人员通过`pnpm changeset`创建变更集文件（.changeset/*.md），声明变更类型（major/minor/patch）和描述
2. **版本PR**：Changesets Bot自动分析并创建版本MR（changeset-release/master），自动更新版本号和CHANGELOG，支持依赖版本联动更新
3. **自动发布**：开发自动版本合并Bot（merge-versions-mr.mjs）
   - 等待Pipeline成功 → 冻结分支 → 合并MR → 发布到NPM → 创建Git Tag → 解冻分支
   - 全流程自动化，零人工干预

**语义化版本规范：**
- 遵循Semantic Versioning（major.minor.patch）
- 支持正式发布和快照发布（snapshot版本：1.0.0-snapshot-20260116）
- 支持预发布测试和快速回滚

##### 5. GitLab MCP工具开发 🔥 自动化提效

**工具定位：**
- 基于标准MCP协议开发的GitLab自动化工具
- 通过Cursor IDE调用，实现工作流自动化

**核心功能：**
1. **智能代码审查**：
   - analyze_mr_changes：深度分析MR变更和diff数据
   - filter_reviewable_files：根据配置规则过滤需要审查的文件
   - get_file_code_review_rules：根据文件类型获取审查规则
   - push_code_review_comments：推送行内评论到GitLab
   - 支持critical/warning/suggestion三级分类
   - 自动生成代码审查报告

2. **MR自动化管理**：
   - get_merge_request：获取MR详细信息
   - get_merge_request_changes：获取文件变更和diff
   - update_merge_request_description：更新MR描述
   - list_merge_requests：列出项目的MR列表

3. **辅助功能**：
   - get_file_content：获取项目文件内容
   - debug_mr_sha_info：调试MR版本信息

**技术实现：**
- 使用TypeScript + @modelcontextprotocol/sdk + @gitbeaker/rest
- stdio传输方式，完美兼容Cursor、Claude Desktop
- 支持多GitLab实例，通过环境变量配置

**提效价值：**
- MR描述生成：从手动编写（20分钟）→ AI生成（2分钟）
- 代码审查：从人工审查（1小时）→ AI辅助（15分钟）
- 技术方案生成：从手动编写（2小时）→ AI生成（20分钟）
- 个人及团队研发效率提升40%+

# 在线文档

---

## 一、协同编辑与数据一致性 (The Core of Collaboration)

在线文档最难的不是「编辑」，而是「多人同时编辑」后的数据同步。

### 1. OT (Operational Transformation) 算法

- **核心原理**：将每一个操作转化为原子指令（如 `Insert(index, char)`）。当 A 和 B 同时操作时，服务端作为中控，对冲突指令进行「变换」。
- **面试重点**：服务端收到指令后，会对比当前的 Version，如果版本滞后，则执行路径合并（Path Merging）。

### 2. CRDT (Conflict-free Replicated Data Types)

- **趋势**：无中心化的数据结构。每个字符都有全局唯一且可排序的 ID。
- **优势**：原生支持离线编辑和 P2P 同步，不需要依赖中心服务器进行变换，数据最终一致性更强。

### 3. 钉钉场景思考

AI 也是一个「虚拟协作者」。当 AI 在改写段落时，用户正在删除该段落，系统如何处理？通常采用**范围锁 (Range Lock)** 或将 AI 指令也接入 OT 链路中。

## 二、长文档 / 大表格渲染优化 (Performance Engineering)

钉钉表格动辄数万行，文档动辄万字，直接渲染会导致页面崩溃。

### 虚拟列表与层级渲染

- **表格**：只渲染视口（Viewport）内的单元格。滚动时，动态回收上方 DOM，复用到下方。
- **Canvas 引擎**：钉钉表格底层大量使用 Canvas 绘制，因为 Canvas 不受 DOM 节点数量限制，且能实现像素级的渲染控制。

### 分片解析 (Incremental Parsing)

将文档数据结构（如 JSON Tree）拆分为小块。在空闲时间（`requestIdleCallback`）进行增量解析，避免长耗时任务阻塞主线程（Lighthouse 指标中的 TBT 优化）。

### 协同状态下的局部更新

当收到其他人的协同指令时，只重新渲染受影响的「行」或「段落」，而非重绘整页。

## 三、从「生成」到「编辑」的逻辑 (AI-to-Editor Pipeline)

这是 AI 文档最关键的交互链路：**如何将 AI 生成的字符串平滑地转化为编辑器可操作的节点？**

### 1. 指令化转换 (Text to Ops)

不要直接把 AI 生成的 HTML 塞进 DOM。应通过 Parser 将流式返回的 Markdown/Text 实时转化为**编辑器内部操作指令（Ops）**。  
例如：AI 输出 `### 标题`，前端解析器识别后立即调用 `editor.insertBlock('heading-3')`。

### 2. 流式状态转换 (Streaming Lifecycle)

- **草稿态 (Drafting)**：AI 生成的内容以「虚影」形式存在，不计入文档版本。
- **确认态 (Commitment)**：用户点击「保留」后，通过一次性事务（Transaction）将所有更改应用到协同链条中。

### 3. 未闭合标签处理

在流式传输中，如果 AI 输出了 `**` 但还没输出结尾的 `**`，前端渲染层需要自动补全，防止后续所有文字都变成加粗状态。

## 四、钉钉场景下的 Agent 落地 (Agentic Workflow)

钉钉 Agent 的核心优势是「**组织内数据互联**」。

### 跨应用编排 (Tool Use & MCP)

- **用户指令**：「根据上周的审批数据生成一份报销汇总表格」。
- **Agent 链路**：调用钉钉审批 API（工具 1）→ 获取 JSON 数据 → 调用表格写入 API（工具 2）。
- **MCP (Model Context Protocol)**：作为标准化层，让 Agent 能以统一接口访问钉钉的考勤、审批、组织架构等异构数据源。

### 人在回路 (Human-in-the-Loop)

在钉钉场景，AI 不能私自修改财务或人事数据。利用 LangGraph 设置 **interrupt** 节点，在执行关键工具前强制暂停，等待用户点击 UI 确认。

### RAG 与企业知识库

利用 RAG 检索企业内部的周报模版、规范文档。通过**语义切片 (Semantic Chunking)** 确保 AI 获取到的文档片段逻辑完整。

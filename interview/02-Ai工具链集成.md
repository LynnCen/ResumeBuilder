## 二、Dify 工作流管线：工具的统一接入层

### 2.1 为什么用 Dify 做工作流管线？

在我们的 AI 工具链中，有多个工具需要接入不同的 AI 模型服务——抠图有专门的分割模型，AI 改图用 inPainting 多模态大模型，AI 扩图用 Outpainting 模型，等等。如果每个工具都直接对接自己的模型接口，会有几个明显的问题：前端代码和 AI 服务高度耦合，换一个模型就得改前端；提示词散落在各处，难以统一管理；新增工具需要修改代码重新部署。

用 Dify 做工作流管线解决了这些问题。Dify 本质上是一个 AI 工作流编排平台，我们在 Dify 上配置每个工具的处理节点——输入格式、模型选择、提示词模板、输出格式——前端只需要和 Dify 的统一 API 交互，不感知底层模型的差异。

### 2.2 工作流管线的具体设计

以 AI 抠图为例，Dify 工作流内部是这样运转的：

**节点一：图像分类。** 接收用户上传的图片，调用图像分类模型判断类型——是人像、商品、还是图形/Logo。这一步很关键，因为不同类型的图片，分割模型的选择和参数配置是不同的，人像分割对发丝处理要求更高，商品分割更关注物体轮廓的清晰度。

**节点二：按类型分流。** Dify 的条件节点根据分类结果路由到不同的分割模型，人像走人像分割模型，商品走商品分割模型，图形走图形分割模型。

**节点三：结果后处理。** 分割模型返回的 mask 数据经过后处理——边缘平滑、羽化处理——最终生成带透明通道的 PNG 图片。

**节点四：返回结果。** 以 SSE 格式流式返回结果，前端实时展示处理进度。

这个设计的核心价值是**工具配置与代码解耦**。运营或算法团队想切换模型、调整提示词，只需要在 Dify 的可视化界面改节点配置，前端零修改，几秒内生效。

### 2.3 统一的工具调用协议

所有 AI 工具走统一的调用接口，参数结构标准化：输入包含工具类型、图片 URL、用户参数（比例、风格、强度等）；输出通过 SSE 流式推送，包含进度信息和最终结果。

前端做了二次封装，对每个工具的特有参数做了类型定义，同时统一处理错误码（余额不足、内容审核拦截、服务超时等），错误情况下展示对应的用户引导提示。

## 三、AI 抠图：前端工程化图像处理

### 3.1 抠图的整体链路

AI 抠图从用户角度看很简单：上传一张图，点击抠图，得到一张透明背景的图。但从技术角度看，这条链路涉及几个关键环节，每一环都有值得说的细节。

**图像上传与预处理**：用户上传图片后，前端先做本地预检——文件大小限制（一般 20MB 以内）、格式校验（JPEG/PNG/WebP 等）、分辨率检查（过大的图片会先做下采样，防止传输慢和服务端超时）。预处理完后将图片上传到 CDN，拿到可访问的 URL，再把 URL 传给 Dify 工作流。

**分割模型处理**：Dify 工作流接收图片 URL，先做图像分类，再调用对应的分割模型。模型返回的核心数据是一张 mask 图——和原图等大的二值图，白色区域表示主体，黑色区域表示背景。

**前端 Canvas 合成**：拿到 mask 后，前端在 Canvas 上做图层合成——将原图和 mask 叠加，只保留 mask 白色区域对应的原图像素，背景区域设置为全透明（alpha=0），最终导出带透明通道的 PNG。

**Matting 精修**：对于头发丝、透明玻璃、毛茸茸的边缘等复杂区域，基础分割的边缘会有锯齿感。我们增加了一个 Matting（精修）步骤，专门处理边缘的半透明过渡，让前景和新背景的融合更自然。这一步可以由用户手动触发（「精修」按钮），或者根据图片复杂度自动触发。

### 3.2 双 Canvas 架构的价值

抠图功能的前端 Canvas 架构分两层。**底层 Canvas** 负责存储原始 mask 数据，作为数据源，不直接展示；**上层 Canvas** 负责交互展示，包括抠图结果预览、用户手动涂抹修正的笔迹、选区边框等。

这样设计的原因是，原始 mask 数据是「静态的」（AI 一次生成），而用户的交互操作是「动态的」（可以不断涂抹修改）。两层分离后，用户每次修改只需要重绘上层 Canvas，不需要重新处理底层 mask 数据，性能更好，逻辑也更清晰。

## 四、SAM 智能选区（重点详解）

### 4.1 SAM 是什么？为什么在项目中集成它？

SAM（Segment Anything Model）是 Meta AI 在 2023 年发布的革命性图像分割模型。它最大的突破在于三个字：**通用性**——不是为特定类别（人、车、猫）训练的专用模型，而是一个能分割「任何东西」的基础模型，对从未见过的对象也能准确分割，具备真正的零样本能力。

在项目中集成 SAM 是为了实现 AI 智能选区功能。用户可以在图片上点击一下，AI 就能精准识别并选中那个区域的完整物体边界——点击商品，自动选中整个商品；点击人脸，自动选中整个人物。配合 AI 改图功能，可以实现「精准换背景」、「局部修图」等高价值操作。

### 4.2 SAM 的工作原理：两阶段分割

SAM 的工作流程分为两个阶段，理解这两个阶段对于理解我们的工程实现非常重要。

**第一阶段：图像编码（Image Encoding）**

将输入图像通过一个强大的视觉编码器（Vision Transformer，ViT-H）处理，生成一个高维的特征向量，称为 **Image Embedding**。这个 Embedding 可以理解为图像的「语义指纹」，包含了图像里每个区域的语义信息——这里是哪只手、哪个轮毂、哪片叶子——维度为 `1 × 256 × 64 × 64`，约 4MB 数据。

这一阶段最重要的特点是：**计算量很大，但只需要执行一次**。一张图片不管用户点击多少次，Embedding 只需要生成一次。这也是我们把这一步放在后端服务器执行的原因——服务器有 GPU 加速，几秒钟能完成。

**第二阶段：提示式解码（Prompt-based Decoding）**

这一阶段在拿到 Embedding 的基础上，根据用户的交互提示（点击的坐标、框选的区域）生成具体的分割结果。解码器相对轻量，关键是它能**在浏览器里实时运行**——我们把 SAM 的解码器导出成了 ONNX 格式，通过 onnxruntime-web 在浏览器中执行推理，每次点击后 50-200ms 就能出结果。

SAM 的解码器每次会输出 **3 个候选 Mask**，并给每个 Mask 打一个质量评分（IOU Prediction），我们从中选质量最高的那个呈现给用户。输出 3 个候选是 SAM 的设计哲学——面对「点击一只手」这种有歧义的输入，3 个候选可能分别是「手指」、「手掌」、「整只手臂」，用户可以从中选择自己想要的。

### 4.3 我们的工程实现：两种分割模式

在工程落地时，我们实现了两种分割模式，适用于不同场景。

**自动分割模式（Auto Mask Mode）**

这种模式的核心思路是「预计算 + 查找」而非「实时推理」。后端 SAM 服务会预先对图片进行全图分析，生成几十到一百个候选区域的 mask 数据，一次性返回给前端。前端把这些候选 mask 都缓存起来，用户每次点击都是在这些预计算好的 mask 中「查找」最匹配的那个，不需要实时调用模型，几乎零延迟。

**交互式分割模式（Interactive Mask Mode）**

当自动分割效果不理想时，切换到交互式模式。用户可以用「正点」（点击包含的区域）和「负点」（点击要排除的区域）精细调整分割边界。每次点击都会触发一次 ONNX 模型的前端推理，利用上一次的结果（predMask）作为输入之一来加速计算，实现增量推理。

对于大多数日常场景，自动分割已经够用且响应极快；对于边界复杂的对象（头发、透明物体、毛绒边缘），交互式分割提供了精确控制的能力。

### 4.4 双 Canvas 架构：数据层与交互层分离

智能选区功能的 Canvas 架构是整个功能中工程设计最精妙的部分，也是面试中最值得详细讲的地方。

我们用了两个 Canvas 承担不同职责：

**Pick Canvas（隐藏拾取层）**

这是一个用户完全看不到的「颜色索引画布」。初始化时，系统为后端返回的每一个 mask 区域分配一个唯一的颜色 ID（比如第 1 个 mask 对应 RGB=(0,0,1)，第 2 个对应 RGB=(0,0,2)，以此类推），然后把所有 mask 区域用各自的颜色 ID 画到这个 Canvas 上。

用户鼠标悬停时，系统读取鼠标坐标对应的 Pick Canvas 上的像素颜色，通过颜色反查出是第几个 mask，O(1) 的时间复杂度完成区域命中检测，比遍历所有 mask 判断是否包含该坐标要快很多。

**Selection Canvas（展示交互层）**

这个 Canvas 叠加显示在图片上方，用于渲染用户看到的所有视觉反馈——悬停时的高亮预览、点击后的选区边框、多选时的区域合并效果。

两层分离的好处是：数据层（Pick Canvas）只在初始化时建立一次，之后所有交互都只是「查表 + 更新展示层」，不需要重新处理 mask 数据，交互极度流畅。

### 4.5 RLE 编码：高效传输的关键

后端 SAM 返回的 mask 数据并不是原始的像素图，而是经过 **RLE（Run-Length Encoding，游程编码）**压缩的格式。这里值得详细解释一下，因为很多面试官会追问数据格式。

Mask 本质上是一张和原图等大的二值图——每个像素只有 0（背景）或 1（前景）两种值。如果是一张 1000×1000 的图片，直接传像素数据需要 100 万个数值，网络传输很慢。而 RLE 的思路是「记录连续相同值的长度」，比如一行像素是「100 个 0，20 个 1，50 个 0，30 个 1」，RLE 只需要存 `[100, 20, 50, 30]` 4 个数字。对于分割 mask 这种大面积连续区域的二值图，RLE 的压缩比通常能达到 10 到 100 倍，显著减少了网络传输量。

前端收到 RLE 数据后，需要解码还原为像素 Canvas 用于渲染。解码逻辑是：按照 RLE 数组依次填充像素，偶数索引是 0 的个数（跳过），奇数索引是 1 的个数（填充白色像素）。

还有一个细节：SAM 输出的 RLE 坐标系是旋转 90 度的（列优先而非行优先），前端解码后需要通过 Canvas 的旋转变换恢复到正常方向，这个细节如果不知道，解码出来的 mask 会出现图像旋转的 bug。

### 4.6 颜色 ID 拾取系统：O(1) 区域命中

这是 Pick Canvas 背后的核心算法，值得专门展开讲。

**问题背景**：假设后端返回了 50 个候选 mask 区域，用户鼠标移动到某个点时，如何快速找到这个点属于哪个 mask？

**朴素方法（低效）**：遍历所有 50 个 mask，对每个 mask 读取该坐标的像素，看是否为前景（alpha > 0）。问题是：50 次 Canvas.getImageData 调用性能很差，而且用户 mousemove 是高频事件，会严重卡顿。

**颜色 ID 方法（高效）**：预先为每个 mask 分配一个唯一颜色，把所有 mask 都画到同一个 Pick Canvas 上（每个 mask 用自己的颜色）。用户鼠标移动时，只读取 Pick Canvas 上鼠标坐标处的颜色，就能 O(1) 确定是哪个 mask。

颜色 ID 的生成很简单：用一个从 1 递增的计数器，把计数值拆成 R、G、B 三个字节，就能唯一表示 16,777,216 个不同区域，远超实际需要。读取颜色后，反向还原出计数值，就找到了对应的 mask。

这个方案巧妙地把「几何包含判断」转化为「颜色查找」，把 O(n) 复杂度降为 O(1)，是计算机图形学中一个很经典的思路（和 GPU 颜色拾取 / Color Picking 是同样的思想）。

### 4.7 图层合成：正选与负选

用户选区时支持「加选」（继续点击添加新区域）和「减选」（点击已选中区域内部的子区域，将其排除）。这个功能的实现依赖 Canvas 的混合模式（globalCompositeOperation）。

正选（包含这个区域）用 `source-over` 模式，把新区域的 mask 叠加到现有结果上——在当前结果基础上增加像素。

负选（排除这个区域）用 `destination-out` 模式，把指定区域的 mask 从现有结果中「扣除」——在当前结果上移除这些像素。

系统会自动判断用户点击的意图：如果点击位置在一个已选中区域的内部，且点击的子区域比父区域小，则自动切换为负选模式，实现「在选中人物后，点击衣服把衣服扣除」这种自然操作。

### 4.8 内阴影描边算法：高亮边框的视觉效果

用户鼠标悬停时，被预览的 mask 区域会显示一圈发光的高亮边框，这是整个功能中视觉感最强的地方，背后的实现也很有意思。

**为什么不用 Canvas 的 strokeRect 或轮廓描边？** 因为 mask 的形状是任意多边形，不是规则形状，用像素逐一描边性能差，而且锯齿明显。

**我们用的方案：Canvas 内阴影（inset shadow）**

具体步骤是：先在一个临时 Canvas 上把 mask 区域填充为纯色；然后利用 Canvas 的 `shadowColor`、`shadowBlur`、`shadowOffsetX/Y` 属性，结合 `destination-in` 混合模式，制造只在区域内侧出现的发光效果。这个技巧的原理是：先用目标 mask 作为「剪切蒙版」，再画一个略大的填充形状，利用阴影向内扩散的视觉效果，模拟出「内阴影描边」的外观。

这个方案不需要手动计算轮廓点，纯 CSS/Canvas 属性实现，渲染质量高（自带 Gaussian 模糊，边缘柔和），性能也好。

### 4.9 ONNX Runtime Web：浏览器内 AI 推理

在交互式分割模式下，SAM 的解码器是直接在浏览器里跑的，这里涉及 ONNX Runtime Web。

**ONNX** 是一种开放的模型格式，各种框架（PyTorch、TensorFlow 等）训练的模型都可以导出成 ONNX 格式。**ONNX Runtime Web** 是微软开源的在浏览器端运行 ONNX 模型的库，底层可以利用 WebAssembly（在所有浏览器运行）或 WebGPU（在支持的浏览器里利用 GPU 加速）。

SAM 的解码器（Mask Decoder）参数量相对较小，导出的 ONNX 文件约 40MB，首次加载耗时 2-5 秒，之后全局缓存，后续使用无感知。每次用户点击触发一次推理，输入是 Image Embedding + 点击坐标，输出是 3 个候选 mask + 质量评分，推理耗时约 50-200ms，交互感良好。

### 4.10 性能优化亮点

**Embedding 复用**：同一张图片不管用户点击多少次，Image Embedding 只生成一次（后端计算一次，结果缓存起来）。不同工具切换到同一张图时，Embedding 也从缓存里取，避免重复请求。

**渐进式功能开放**：用户切换到 SAM 工具后，系统先立刻展示自动分割结果（只需要加载后端预计算的 mask 数据，很快），同时在后台异步加载 ONNX 模型和 Embedding。ONNX 模型和 Embedding 都准备好后，再开放交互式分割的精细控制功能，用一个提示引导用户。

**mousemove 防抖**：颜色 ID 拾取虽然是 O(1)，但 mousemove 频率很高（每秒 60 次以上），每次都要读 Canvas 像素。我们加了约 16ms 的节流（对应 60fps），既保证流畅感，又不浪费计算。

**历史记录的增量利用**：在交互式模式中，多次点击的历史会被记录。新点击时，如果新点击序列的前缀和历史匹配（用户撤销了最后一次点击，然后点了个新点），系统会复用之前的中间计算结果（predMask），不需要从头推理，降低计算量。

## 五、WASM 图片压缩：纯前端高性能处理

### 5.1 为什么用 WASM？

图片压缩是一个计算密集型的操作，如果放在服务端处理，需要上传原图等待处理再下载，来回网络传输对用户体验影响很大。如果用纯 JavaScript 实现压缩算法，性能不够。

WebAssembly（WASM）是一种可以在浏览器中运行的二进制格式，接近原生速度，填补了「浏览器需要高性能计算」的空白。我们把两个成熟的 C++/Rust 压缩算法编译成 WASM 在浏览器运行，实现了真正的纯前端处理——数据不离开用户设备，隐私更安全，速度也更快。

### 5.2 两个 WASM 算法模块

**MozJPEG（C++ 编译）**：Mozilla 基金会开源的 JPEG 编码器，是标准 libjpeg 的优化版本，相同质量下体积比原生 JPEG 小 10-20%。主要用于照片类图片的压缩。压缩率可以通过 quality 参数控制（0-100），90 的质量对应肉眼几乎无损，文件大小通常减少 40-60%。

**ImageQuant（Rust 编译）**：高质量的 PNG 量化压缩库，使用感知色彩量化算法，将 24 位真彩色 PNG 转换为 8 位索引色 PNG，在保持视觉质量的前提下大幅缩小文件体积。对于那些颜色较少的图形、插图、截图类内容，压缩效果极佳，能减少 60-80% 的体积。

### 5.3 工程实现要点

**异步加载，按需使用**：两个 WASM 模块都是按需加载的，用户打开压缩工具时才开始下载和初始化，不影响首屏性能。WASM 文件本身有缓存，第二次使用无需重新下载。

**Worker 线程隔离**：压缩运算在 Web Worker 线程里执行，不阻塞主线程，UI 保持响应。主线程只负责传入图片数据、接收压缩结果，运算过程不影响用户继续操作其他功能。

**自动格式选择**：对于 JPEG 格式的图片走 MozJPEG，对于 PNG 格式的图片，系统会先分析色彩复杂度——如果颜色数量少（适合量化），走 ImageQuant；如果是真彩色照片（量化会损失质量），走先转 JPEG 再压缩的方案。这个判断逻辑帮用户自动选择最优压缩路径。

### 5.4 压缩效果

实际使用数据：平均压缩率 80% 以上（原图 5MB 压缩后约 1MB），对于图形类 PNG 最高可达 90% 以上。整个过程在本地完成，无网络传输延迟，1MB 的图片通常在 500ms 内完成压缩。

## 六、AI 视频生成：长时任务的状态管理

### 6.1 三种视频生成模式

AI 视频生成从 0 到 1 由我独立负责落地，实现了三种模式：

**文生视频**：用户输入文字描述，AI 根据描述生成视频内容。核心参数包括时长（3 秒 / 5 秒 / 10 秒）、画面比例、风格代码等。

**图生视频**：用户提供一张图片作为首帧，AI 生成让这张图「动起来」的视频。可以指定运镜方式（推进、拉远、旋转等），让静态图产生动态效果。

**视频特效**：对已有视频应用风格化效果，比如转换为动漫风格、水墨风格等。

### 6.2 长时任务的核心挑战

视频生成和图片生成最大的区别是**时间**——图片生成通常 5-10 秒，视频生成通常 30-120 秒，最长可能超过 3 分钟。这带来了两个工程挑战：

**挑战一：连接超时**。维持一个长达几分钟的 SSE 或 HTTP 连接不现实，会受到服务器、代理、CDN 等各层的超时限制。

**挑战二：进度反馈**。用户面对一个几分钟的等待界面，如果没有进度指示，体验极差，很可能直接关闭页面。

### 6.3 解决方案：提交分离 + 智能轮询

**提交与等待分离**：用户提交视频生成任务后，服务端立刻返回一个 taskId，连接随即关闭。这一步是「异步提交」，客户端拿到 taskId 就算提交成功，接下来服务端异步处理，不需要保持连接。

**前端主动轮询**：客户端用 taskId 定时查询任务状态接口，获取当前进度（0% → 30% → 60% → 100%）和预计剩余时间。这个「客户端主动轮询」的模式，比让服务端主动推送更简单，也更稳定。

**指数退避算法**：轮询间隔不是固定的，而是动态调整的。初始间隔 3 秒，如果连续几次返回「处理中」，间隔逐渐延长——3 秒、6 秒、12 秒……最长不超过 30 秒。这个策略的好处是：任务初期（进度变化快）频繁查询保证及时更新；任务后期（变化慢）降低频率减少服务器压力。

**超时熔断机制**：轮询最多持续 2 分钟（40 次查询），超出后自动停止，向用户展示「生成时间较长，可以稍后查看历史记录」的提示，避免无限等待浪费资源。用户也可以手动停止轮询，任务依然在后台继续，生成完成后会在历史记录里展示。

### 6.4 进度展示设计

视频生成期间的 UI 设计也有一些值得说的考量。进度条是「真实进度」而非「假进度」——数值直接来自服务端的实时进度，不是前端用动画模拟的。这样用户看到 60% 就是真的处理了 60%，预计剩余时间更准确，不会有「一直卡在 99%」的糟糕体验。

服务端进度的计算依据是视频帧的生成状态：模型按帧生成视频，完成多少帧 / 总帧数 = 当前进度。

## 七、高频面试题 Q&A

### Q1：SAM 和普通的图像分割模型有什么本质区别？

**A**：本质区别在于通用性和交互范式。

普通的图像分割模型（比如人像分割、商品分割）是有监督训练的，针对特定类别——训练时见过什么就能分割什么，对从没见过的新类别效果很差，而且每个应用场景可能需要一个专用模型。

SAM 是一个基础模型（Foundation Model），它的训练数据包含了超过 11 亿个 mask，覆盖了极其多样的对象类别，从而具备了真正的「零样本」泛化能力——即使对从未见过的对象类型，也能准确分割。

另一个本质区别是交互范式。SAM 是「提示驱动」的，用户给一个点、一个框、甚至一个粗略的 mask 作为提示，模型就能生成精确的分割结果，而且可以迭代修正。这种人机协作的方式比完全自动的分割更灵活，更符合设计类工具的使用场景。

### Q2：两阶段分割为什么这样划分？前端和后端各做什么？

**A**：划分依据是计算量和实时性要求。

Image Encoder（图像编码器）是 SAM 中计算量最大的部分，参数约 600M，需要 GPU 才能在可接受的时间内完成（通常 1-3 秒）。但它的输出 Embedding 对同一张图片是固定的，不随用户交互变化。所以这一步放后端做：服务器用 GPU 计算，结果存到 OSS，一张图只算一次，前端请求拿到 URL 后下载使用。

Mask Decoder（掩码解码器）参数量小很多，计算轻量，设计之初就考虑了在浏览器运行的场景。它需要对用户每次点击实时响应（<200ms），网络往返（至少 100ms 以上）会让体验很差，所以这一步在前端用 ONNX Runtime Web 本地运行，真正做到即时响应。

这个前后端分工是一个很好的「按职责分配计算资源」的工程决策：重计算 + 一次性 → 后端；轻计算 + 高频次 → 前端。

### Q3：颜色 ID 拾取系统，为什么不直接遍历所有 mask 判断？

**A**：核心是性能问题，在高频 mousemove 场景下更明显。

遍历方法需要对每个 mask 做一次 Canvas.getImageData 读取，如果有 50 个 mask，就是 50 次 DOM 操作。Canvas.getImageData 是有性能代价的，因为需要从 GPU 内存把数据读回 CPU 内存，这个操作本身就慢。再加上 mousemove 每秒触发几十次，遍历方法会导致明显卡顿。

颜色 ID 方法把「判断属于哪个 mask」转化为「读取一次颜色 + 查哈希表」，只有一次 Canvas 读取操作，时间复杂度 O(1)。这个思路来源于计算机图形学里的 Color Picking 技术，在 3D 渲染里经常用来实现「点击选中物体」——给每个物体赋一个唯一颜色，用不可见的 pick pass 渲染一遍，然后读取点击位置的颜色来确定选中的是哪个物体。

### Q4：RLE 编码解码有什么坑？

**A**：有两个坑，都是实际遇到过的。

第一个坑是**坐标系问题**。SAM 输出的 RLE 数据是按列优先顺序编码的，也就是先填完第一列，再填第二列，而通常图像处理是行优先的（先填第一行，再填第二行）。不了解这个的话，解码出来的 mask 会被旋转 90 度，图像形状是对的，但方向不对。解决方法是先按列优先解码到临时 Canvas，再旋转 90 度还原正常方向。

第二个坑是**大图性能**。4K 图片的 mask 数据解码涉及到对 400 万像素的操作，用 JavaScript 的 for 循环逐像素操作会很慢。我们用 ImageData 的 TypedArray 批量操作，以及 OffscreenCanvas 放到 Web Worker 里异步处理，避免主线程卡顿。

### Q5：WASM 和普通 JavaScript 在图片压缩上的性能差距有多大？

**A**：实际测试差距显著，尤其是对于计算密集的压缩算法。

以 2MB 的 JPEG 图片重新编码为例，纯 JavaScript 实现（canvas.toBlob）底层用的是浏览器内置的 JPEG 编码器，压缩质量有限，通常需要 200-500ms。MozJPEG 的 WASM 版本会慢一些（首次初始化加 WASM 加载约 1-2 秒），但实际压缩过程在 WASM 里运行，速度和压缩质量都比原生 JavaScript 好——相同质量等级下，MozJPEG 生成的文件比浏览器内置编码器小 15-25%，而且是在不阻塞主线程（Worker 里运行）的前提下完成的。

对用户来说感知最明显的是结果质量：同样压缩到 500KB，MozJPEG 的视觉质量明显好于 canvas.toBlob，细节保留更多，高频纹理更清晰。

### Q6：视频生成的指数退避轮询，如果用户网络不好怎么办？

**A**：我们在轮询逻辑里加了网络错误的专门处理。

正常情况下，轮询失败（服务端返回非 200，或任务状态是 failed）会终止轮询并提示用户。但对于网络请求本身失败的情况（fetch error，通常是网络波动），我们不会立刻终止，而是做一次本地重试等待（间隔加倍后再试），连续 3 次网络错误才真的停止。

另外，为了应对用户中途关闭页面然后再打开的情况，我们把进行中的任务 taskId 存到 localStorage。用户重新打开工具时，如果检测到有未完成的任务，会自动恢复轮询，无感知地继续追踪进度。

### Q7：SAM 的前端 ONNX 推理，在低端设备上会不会很卡？

**A**：会，这是我们做过针对性优化的地方。

在配置差的机器上，每次推理可能需要 500ms 到 1 秒，用户点击后有明显的等待感。我们的应对策略：

第一，**优先推荐自动分割模式**。自动分割是查表操作，几乎无推理，在所有设备上都流畅，我们把它作为默认模式，交互式分割作为「进阶选项」。

第二，**设备能力检测**。通过简单的计算密集型操作，在用户首次使用时评估设备计算能力，低端设备上隐藏「交互式精细分割」入口，避免用户点击后卡顿。

第三，**WebGPU 优先**。ONNX Runtime Web 支持多种后端：WebGPU（最快，利用 GPU）、WebAssembly（次之，多线程 WASM）、CPU（最慢，纯 JS）。我们按优先级依次尝试，优先用 WebGPU，不支持则降级。在支持 WebGPU 的现代浏览器上，推理时间能从 200ms 缩短到 30-50ms。

## 一、为什么需要无限画布

### 1.1 传统 DOM 渲染的瓶颈

在设计编辑器中，用户可能会在一个画布里放置数百甚至数千个元素——AI 生成图、背景图、文字、矢量图形等。传统方案是将每个元素渲染为一个 DOM 节点，在元素较少时表现尚可，但当元素数量增长之后，问题就会集中爆发：

首先是**布局性能问题**。浏览器每次更新一个 DOM 节点的位置或样式，都可能触发整个文档的重排（Reflow）和重绘（Repaint）。当画布上有一千个元素时，哪怕只拖动其中一个，浏览器也可能要重新计算整棵 DOM 树，导致明显的帧率下降。

其次是**内存压力**。每个 DOM 节点除了持有业务数据外，还要维护浏览器内部的大量样式信息、事件监听、无障碍树等元数据，内存占用远超元素本身。一旦元素数量超过阈值，页面会出现明显的卡顿甚至崩溃。

最后是**缩放体验差**。设计类产品需要支持从 10% 到 400% 的任意缩放，DOM 方案要么用 CSS transform 缩放整个容器（会引发子级模糊、点击区域错位等问题），要么对每个元素单独计算样式（计算量极大），两条路都走不通。

### 1.2 WebGL（PixiJS）方案的优势

我们选择以 PixiJS 为底层渲染引擎，将整个画布绘制到一块 `<canvas>` 元素上，再在其上叠加轻量级的交互层。这一方案从根本上解决了上述问题：

**硬件加速**。所有绘制操作通过 WebGL 指令提交给 GPU 并行处理，CPU 的压力大大减轻，画布渲染轻松保持 60fps。

**虚拟化渲染**。渲染引擎只为当前视口内可见的元素创建渲染对象。当用户滚动或平移画布时，视口外的元素不会参与绘制计算，其纹理也会被及时回收，将内存和 CPU 用在刀刃上。

**批量 Draw Call**。PixiJS 会将使用同一纹理的多个 Sprite 合并成一次 Draw Call 提交给 GPU，把原本数百次的 CPU-GPU 通信压缩到个位数，大幅提升渲染吞吐量。

## 二、整体架构概览：三层模型

无限画布渲染引擎采用了清晰的三层架构，从上到下依次是：**桥接层（Bridge Layer）**、**视图模型层（VM Layer）**、**引擎层（Engine Layer）**。

```
┌─────────────────────────────────────────────────────┐
│                  Bridge Layer（桥接层）               │
│  监听编辑器 Model 的变更事件 → 将语义转换为渲染指令    │
│  → 通过 commit(action) 驱动下层更新                  │
└─────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────┐
│                  VM Layer（视图模型层）               │
│  VmEngine 管理所有 ViewModel（ImageVm / TextVm 等）  │
│  ViewModel = 状态管理 + 生命周期钩子 + 纹理/排版控制  │
└─────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────┐
│                  Engine Layer（渲染引擎层）            │
│  PixiJS WebGL：Sprite / Container / Texture / Stage │
│  Ticker 驱动每帧绘制，输出到 HTMLCanvasElement        │
└─────────────────────────────────────────────────────┘
```

这三层的分工体现了一个核心设计原则：**数据与渲染彻底分离**。业务数据只存在于编辑器的 Model 层；渲染侧不持有任何业务数据，只响应 Model 的变更来更新视图。

### 2.1 Surface：渲染侧的顶层调度者

在具体实现上，渲染侧还有一个更顶层的调度者——**Surface**（画布表面）。Surface 本身不直接绘制任何东西，而是协调下属三个组件共同工作：

- **Viewport（视口）**：管理相机，负责缩放、平移，以及屏幕坐标和画布坐标之间的双向转换。
- **Processor（处理器）**：接收来自桥接层的 commit 指令，调用 VmEngine 增删改对应的 ViewModel。
- **PluginSystem（插件系统）**：以插件方式扩展画布能力，如背景渲染、网格线、选中框控制点、对齐辅助线等。

```
                       Surface
                      /  |  \
              Viewport Processor PluginSystem
                          |
                       VmEngine
                      /   |   \
               ImageVm  TextVm  GroupVm ...
                          |
                       PixiJS Stage
```

## 三、数据通信全链路

理解整个系统最关键的一点，是理解从"用户操作"到"画布更新"的完整数据流。整条链路可以分为七个步骤：

### 第一步：业务操作，创建 Model

用户的所有操作（拖入图片、添加文字、删除元素）都通过编辑器提供的 API 发起。以添加图片为例，调用 `editor.addElement(imageModel, parentLayout)` 会将一个 `ImageElementModel` 对象插入到父节点（Layout）的 `elements` 数组中。

这个 Model 是一个普通的 TypeScript 对象，包含了图片的 URL、宽高、坐标、变换矩阵、滤镜列表等一切业务信息。Model 只存在于编辑器侧，渲染侧不直接持有它。

### 第二步：makeSnapshot，记录操作

`addElement` 完成后，编辑器内核会调用 `makeSnapshot` 记录这次操作，将其转换为一个结构化的 `action` 对象（如 `{ tag: 'add_element', element, parent, index }`），并通过 YJS 写入协同文档（YDoc）。

YDoc 的更新完成之后，编辑器内核会派发一个 `base.afterApply` 事件，携带这个 action 对象。**这个事件是编辑器侧与渲染侧之间唯一的通信桥梁**，语义是"数据已经更新完毕，渲染侧可以安全地同步了"。

### 第三步：桥接层监听，转发 commit

渲染侧的桥接层（core-plugin 中的 `useCommit`）在初始化时订阅了 `base.afterApply` 事件。当事件到来时，它将编辑器语义（`add_element`）翻译成渲染语义，并调用 `surface.commit(action)` 向渲染侧提交指令。

这一步是两侧解耦的关键：编辑器侧不知道渲染侧是 PixiJS 还是其他引擎；渲染侧也不知道业务逻辑，只处理规范化的 action 指令。

### 第四步：Processor 处理指令

Surface 接到 commit 调用后，将 action 转发给 Processor。Processor 根据 action 类型（`add_element` / `update_element` / `remove_element`）调用 VmEngine 的对应方法。

以 `add_element` 为例，Processor 会调用 `vmEngine.buildVm(element)`，让 VmEngine 为这个 Model 创建一个对应的 ViewModel，然后挂到父 ViewModel 下面。

### 第五步：VmEngine 创建 ViewModel

VmEngine 是整个渲染侧的核心调度器，它维护着一张以 `model.uuid` 为键的 ViewModel 缓存表（`elementMap`）。

当需要为某个 Model 创建 ViewModel 时，VmEngine 首先检查缓存，若存在则直接复用；若不存在，则根据 `model.type` 实例化对应的 ViewModel 类。图片对应 `ImageVm`，文字对应 `TextVm`，分组对应 `GroupVm`，以此类推。

ViewModel 创建完成后，VmEngine 调用其 `init(model, context)` 生命周期方法，触发异步资源加载（图片的纹理加载、文字的字体加载与排版）。

### 第六步：ViewModel 异步加载资源，更新 PixiJS 对象

以 `ImageVm` 为例，`init` 阶段会从 Model 中取出图片 URL，委托 `TextureManager` 异步加载纹理。`TextureManager` 内部有完整的缓存机制和等待队列——同一张图片即使被多个元素同时请求，也只会发出一次网络请求，加载完成后统一通知所有等待者。

纹理加载完成后，`ImageVm` 将纹理绑定到自己持有的 PixiJS Sprite 对象，并调用 `setState` 更新内部状态，触发 `render()` 方法更新 Sprite 的位置、尺寸、变换矩阵等属性。

以 `TextVm` 为例，`init` 阶段会调用 TypeTool 排版引擎对文字内容进行排版计算，得到每个字符的坐标、字形信息，然后用 Canvas 2D 或 Skia 后端绘制成位图，再作为纹理挂到 PixiJS 显示对象上。

### 第七步：PixiJS Ticker 驱动渲染

PixiJS 内部有一个 Ticker（定时器），以 `requestAnimationFrame` 为基础，每帧（约 16ms）调用一次 `renderer.render(stage)`。渲染时，引擎会遍历整个显示对象树，对每个可见且在视口内的节点调用其 `_render` 方法，将顶点数据批量提交给 GPU，最终绘制到 `<canvas>` 上。

## 四、各层核心职责详解

### 4.1 桥接层的职责

桥接层是编辑器侧和渲染侧之间的"翻译官"。它的存在使得两侧可以完全独立演进：只要双方都遵守约定的 action 格式，编辑器不需要了解 PixiJS，渲染侧也不需要了解业务逻辑。

桥接层处理三种核心 action：
- `add_element`：通知 VmEngine 创建新的 ViewModel 并挂到场景树
- `update_element`：通知对应 ViewModel 重新读取 Model 并更新渲染
- `remove_element`：通知 VmEngine 销毁 ViewModel，释放纹理和内存

### 4.2 VM 层的职责

ViewModel 层是整个系统中最复杂的部分，承担着三重职责：

**状态管理**：每个 ViewModel 持有一份内部状态对象 `_state`，通过 `setState()` 更新。`setState` 内部会触发生命周期方法链：`shouldUpdate` 判断是否需要更新、`beforeUpdate` 执行更新前处理、`render` 执行实际渲染、`afterUpdate` 执行收尾工作。这套机制避免了不必要的重复渲染。

**生命周期管理**：ViewModel 有两套生命周期。数据模型生命周期（`modelCreated / modelUpdated / modelRemoved`）响应来自 VmEngine 的数据变更通知；视图模型生命周期（`shouldUpdate / beforeUpdate / render / afterUpdate`）控制内部渲染流程。

**三层图层结构**：每个 `BaseElementVm` 的 PixiJS 容器内部划分为三个子层：`contentLayer`（渲染自身内容，如图片纹理或文字位图）、`childrenLayer`（渲染子元素，用于分组场景）、`watermarkLayer`（渲染版权水印）。三层分离使得不同类型的内容可以独立控制，互不干扰。

### 4.3 引擎层的职责

引擎层由 PixiJS 提供，主要包含以下核心概念：

**DisplayObject**：所有可见元素的基类，持有坐标、旋转、缩放、透明度等基础属性。

**Container**：可以包含子元素的容器节点，形成树形的场景图（Scene Graph）。

**Sprite**：用于显示纹理的叶子节点，也是最常用的显示对象。

**Texture / BaseTexture**：`BaseTexture` 代表 GPU 显存中的实际纹理资源，对应一张图片或一块 Canvas；`Texture` 是 `BaseTexture` 的视图，可以只截取其中一个区域，多个 `Texture` 可以共享同一个 `BaseTexture`，避免重复占用显存。

## 五、关键技术概念

### 5.1 虚拟化渲染

虚拟化渲染是支撑"无限画布"的核心优化手段。其思路与 React 虚拟列表一致：**只渲染用户当前能看到的元素，视口外的元素不创建渲染对象**。

具体实现上，每个 `CanvasSprite`（我们对 PixiJS Sprite 的扩展基类）在 `_render` 方法中会先调用 `renderer.frame.intersects(bounds)` 判断自身是否与当前视口相交。如果不相交，直接 return，不执行任何绘制；如果相交，才进入正式的纹理更新和绘制流程。

当视口移动（用户滚动、平移画布）时，渲染引擎会动态发现新进入视口的元素并触发其资源加载，同时对离开视口超过一定时间的元素释放其纹理，将显存还给系统。

### 5.2 动态精度（LOD）

在不同缩放比例下，元素所需的纹理精度是不同的：缩小到 10% 时一张 4K 图只占屏幕上几个像素，完全不需要加载高精度纹理；而放大到 400% 时用户需要看到清晰的细节。

`DynamicSprite`（`CanvasSprite` 的子类）在每帧渲染时会检测当前的缩放比例，如果精度变化超过阈值，就标记自身为"脏"状态，在下一帧重新按当前精度生成纹理。这种"用即生成，缩即降级"的策略在清晰度和性能之间取得了良好平衡。

### 5.3 快照机制（Snapshot）

当元素刚进入视口、高精度纹理还在异步加载时，渲染引擎会先用一张 16×16 像素的半透明占位纹理（Snapshot）代替显示，避免页面上出现空白区域。这种"先出现、后清晰"的渐进式加载体验，用户感知上比空白等待要好得多。

### 5.4 纹理缓存与等待队列

`TextureManager` 内部维护了两层缓存机制。一是已加载成功的纹理缓存（`BaseTexture.cache`），相同 URL 的图片不会重复请求；二是加载中的等待队列（`waitingMap`），同一 URL 同时被多个元素请求时，只发一次网络请求，加载完成后统一分发给所有等待方。这两个机制结合，有效避免了重复加载和竞态问题。

## 六、完整示例：添加一张图片的全流程

以用户将一张 AI 生成的图片拖入画布为例，梳理从用户操作到画面显示的完整链路：

**第一步**，业务侧调用 `editor.createElement({ type: 'image', url: 'xxx', width: 400, height: 300 })` 创建 `ImageElementModel`，再调用 `editor.addElement(imageModel, layout)` 将其挂入当前 Layout。

**第二步**，编辑器内核执行 `makeSnapshot({ tag: 'add_element', element: imageModel, parent: layout, index: 5 })`，将操作写入 YDoc，完成后触发 `base.afterApply` 事件。

**第三步**，桥接层的事件监听器收到 `base.afterApply`，提取 action 并调用 `surface.commit({ type: 'add_element', element: imageModel, parent: layout, index: 5 })`。

**第四步**，Surface 将 commit 转发给 Processor，Processor 调用 `vmEngine.buildVm(imageModel)`。

**第五步**，VmEngine 查缓存未命中，根据 `model.type === 'image'` 实例化 `new ImageVm()`，调用 `setModel(imageModel)` 和 `init(imageModel, context)`，然后将其挂到 `layoutVm.childrenLayer` 下，同时以 `imageModel.uuid` 为 key 写入 `elementMap`。

**第六步**，`ImageVm.init` 内部调用 `TextureManager.buildTexture(url)`，发起异步纹理加载。加载期间，画布上显示 16×16 的半透明快照占位。

**第七步**，纹理加载完成，`ImageVm` 将纹理绑定到内部的 `DynamicImage`（Sprite），调用 `setState` 触发 `render()`，更新 Sprite 的宽高、位置、变换矩阵。

**第八步**，PixiJS Ticker 在下一帧执行时，遍历场景图，将 `ImageVm.view`（一个 Container）中的 Sprite 顶点数据打包成批次，通过一次 WebGL Draw Call 提交给 GPU，图片出现在画布上。

## 七、总结

无限画布渲染引擎的核心设计思想可以归纳为以下三点：

**分层解耦**。通过 Bridge → VM → Engine 三层架构，将业务数据、视图逻辑、底层渲染彻底分离。编辑器侧只管理 Model，渲染侧只响应 action 指令，两侧通过事件总线（`base.afterApply`）进行松耦合通信，互不侵入。

**按需渲染**。虚拟化渲染、视口裁剪、纹理生命周期管理等一系列机制，确保系统的计算和内存消耗始终与用户可见的内容成正比，而不是与画布上的总元素数成正比。这是支撑"无限画布"名称的核心能力。

**质量自适应**。动态精度（LOD）和快照机制使得系统能在渲染质量和性能之间动态权衡——用户放大时追求清晰，用户缩小或快速滚动时追求流畅，始终给用户一个响应及时的使用体验。

这套架构设计让编辑器能够在同一个画布上流畅地承载数千个元素，同时保持 60fps 的渲染帧率，这是纯 DOM 方案无法实现的。

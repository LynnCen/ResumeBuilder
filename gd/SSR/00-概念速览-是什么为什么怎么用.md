# SSR 概念速览

> **文档说明**  
> 用最简单的话说明 SSR（服务端渲染）是什么，解决什么问题，水合与前后端同构怎么理解，以及开发时最容易踩的坑和解决方案。适合先通读再深入 01～12 详细文档。

---

## 一、一句话版

| 概念 | 一句话 |
|------|--------|
| **SSR** | 在**服务器**上跑前端框架代码，生成**完整 HTML** 再返回给浏览器，用户先看到内容再加载 JS。 |
| **水合（Hydration）** | 客户端 JS「接管」服务端渲染好的 HTML，绑事件、变可交互；**前提是服务端和客户端首次渲染结果一致**。 |
| **前后端同构** | **同一套**组件代码既在服务端跑一遍（出 HTML），又在客户端跑一遍（水合 + 交互）；写一次，两处用。 |
| **生命周期缓存** | 把**服务端**请求到的数据随 HTML 带给客户端，水合时直接用、**不再请求**，保证两端数据一致，避免水合失败。 |

---

## 二、SSR 是什么、解决什么、怎么用

### 2.1 是什么

**SSR（Server-Side Rendering）**：在服务器端执行前端框架（如 Vue/React）的渲染逻辑，生成**带内容的 HTML**，再把这份 HTML 返回给浏览器。浏览器先展示这份 HTML，再加载你的 JS，由 JS 做「水合」把页面变成可交互的。

### 2.2 解决什么问题

| 问题 | 纯客户端渲染（CSR） | SSR |
|------|---------------------|-----|
| **SEO** | 爬虫拿到的是空壳 HTML | 爬虫能拿到完整内容 |
| **首屏** | 先白屏，等 JS 跑完才出内容 | 首屏即 HTML，更快看到内容（TTFB、FCP、LCP 更好） |
| **体感** | 白屏时间长 | 减少白屏，体感更流畅 |

### 2.3 整体流程（怎么用）

```
用户请求
   ↓
CDN / BFF（鉴权、AB 实验、缓存检查）
   ↓
Meta SSR 服务
   ├─ 执行 Vue/React 渲染
   ├─ 数据请求（含生命周期缓存）
   ├─ 输出 HTML + 内嵌缓存数据
   ↓
浏览器
   ├─ 先显示 HTML（用户已可见）
   ├─ 加载 JS
   ├─ 水合：用内嵌数据、不重复请求，接管 DOM
   └─ 页面可交互
```

---

## 三、水合（Hydration）为什么重要

**水合**：客户端 JS 在已有 HTML 上「挂载」应用，绑定事件、恢复状态，使静态 HTML 变成可点击、可输入的页面。

**关键要求**：

- 服务端和客户端的**首次渲染结果必须一致**（同样的 DOM 结构、同样的文本）。
- 若不一致，框架会报「水合失败」，无法复用服务端 HTML，可能闪烁或报错。

**为什么会不一致？**

- 数据在服务端请求了一次，在客户端又请求了一次，**两次结果不同**。
- 只在客户端用的数据（如 `onMounted` 里 `fetch`），服务端没有，导致服务端渲染的是「空」或「Loading」，客户端渲染的是「有数据」→ 不匹配。

**解决办法**：用**生命周期缓存**——服务端请求到的数据随 HTML 传到客户端，水合时直接用这份数据，不二次请求，保证两端一致。

---

## 四、前后端同构（Isomorphic）是什么

**同构**：同一份组件代码会在两个环境各跑一次：

1. **服务端**：跑一遍，产出 HTML，然后请求结束。
2. **客户端**：再跑一遍，在已有 HTML 上做水合，之后响应用户操作。

因此要注意：

- **环境差异**：服务端没有 `window`、`document`、`localStorage` 等，不能在同构路径里直接用。
- **生命周期差异**：`onMounted`、`onUpdated` 等只在客户端执行；服务端主要执行 `setup` 和 `onServerPrefetch`。
- **不能单例**：若用单例存「当前用户」等状态，多请求会共享同一实例，造成**请求污染**（用户 A 看到用户 B 的数据）。

---

## 五、98% 的 BUG 来自这四类问题

文档总结：SSR 里绝大多数问题来自下面 4 点。

| 问题 | 说明 | 后果 |
|------|------|------|
| **1. 水合失败** | 服务端和客户端首次渲染状态不一致 | 无法复用服务端 HTML，报错或闪烁 |
| **2. 浏览器 API** | 在 SSR 渲染路径里用了 `window`、`document` 等 | 服务端 500 或启动失败 |
| **3. 响应式数据** | 在 SSR 阶段依赖 `watch`、`computed` 等响应式更新 | 服务端只渲染一次，行为与客户端不同，HTML 可能不符合预期 |
| **4. 单例模式** | 用单例缓存用户/请求级状态 | 多请求共享，**请求污染**，用户 A 看到用户 B 的数据 |

**对应原则**：

- 用**生命周期缓存**保证两端数据一致，避免水合失败。
- 访问浏览器 API 放到 **`onMounted`** 或 `import.meta.env.SSR` 判断里。
- 服务端当「一次性渲染」，不依赖响应式更新。
- 按**请求/上下文**创建实例，不用全局单例存用户状态。

## 六、生命周期缓存：解决「数据执行两次、结果不一致」

### 6.1 问题

同构下，数据请求会执行两次（服务端一次、客户端一次）。若两次结果不同（或仅客户端有数据），就会水合失败。

### 6.2 思路：生命周期缓存

把**服务端**请求到的结果，**随 HTML 传到客户端**；客户端水合时**直接用这份数据，不再发请求**。

流程简述：

```
服务端：执行 cacheProvider → 请求数据 → 缓存 → 渲染 HTML → 把缓存序列化进 HTML
客户端：解析 HTML 里的缓存 → 执行 cacheProvider 时直接用缓存（不请求）→ 水合成功
```

### 6.3 怎么用（Vue 3 示例）

```javascript
import { cacheProvider } from '@web-widget/helpers/cache';

// 第一个参数：唯一 cache key（建议带业务含义 + 动态参数）
// 第二个参数：返回 Promise 的请求函数
const data = await cacheProvider('article_123', async () => {
  return await fetchArticle(123);
});
```

- 服务端：执行请求，结果被缓存并写入 HTML。
- 客户端：读到同一份缓存，直接使用，不再次请求，保证首屏数据一致。

## 七、这么做的好处（为什么用 SSR）

| 维度 | 收益 |
|------|------|
| **SEO** | 爬虫拿到完整 HTML，利于收录与排序 |
| **首屏与性能** | 首屏即内容，TTFB/FCP/LCP 更好，白屏时间短 |
| **体感** | 用户更快看到可读内容，再逐渐可交互 |
| **架构** | 同构一套代码，服务端出 HTML、客户端水合，逻辑统一 |

代价是：要处理环境差异、生命周期、请求污染、缓存与鉴权等，本系列文档（01～12）就是围绕这些展开的。

---

## 八、关系小结

| 概念 | 作用 |
|------|------|
| **SSR** | 在服务端生成完整 HTML，解决首屏与 SEO |
| **水合** | 客户端用同一份数据接管 HTML，变为可交互；**依赖首屏数据一致** |
| **同构** | 一套代码跑两遍（服务端 + 客户端），需注意环境与生命周期 |
| **生命周期缓存** | 服务端数据随 HTML 下发给客户端，避免重复请求、保证一致，是避免水合失败的关键 |

**开发时记住**：  
数据用 **cacheProvider** 在服务端取、并带到客户端；**不用**浏览器专有 API（或包在 `onMounted` 里）；**不用**单例存请求级状态；保证**服务端和客户端首次渲染一致**。

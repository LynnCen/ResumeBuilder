# Import Map 与 WPM 概念速览

> **文档说明**  
> 用最简单的话说明 Import Map、WPM 是什么，解决什么问题，如何与构建配合，以及带来的好处。适合先通读再深入 01～04 详细文档。

---

## 一、一句话版

| 概念 | 一句话 |
|------|--------|
| **Import Map** | 浏览器里的一份「包名 → 实际 URL」映射表，让你能写 `import _ from 'lodash'` 而不是一长串 CDN 地址。 |
| **WPM** | 根据项目依赖**自动生成**这份映射表的工具，避免手写、避免和 node_modules 版本不一致。 |

---

## 二、Import Map 是什么、解决什么、怎么用

### 2.1 是什么

Import Map 是浏览器的一个标准（WICG 提案），用一段 JSON 定义：**当代码里写 `import xxx from '包名'` 时，到底去哪个 URL 拉取**。

### 2.2 解决什么问题

- 浏览器默认只认 URL 或相对路径，不认 `'lodash'` 这种「裸模块名」。
- Import Map 给裸模块名配上真实 URL，就能在浏览器里直接写 `import 'lodash'`。
- 顺带还能：用 URL 里的版本号做长期缓存、用 scopes 在同一页面用同一包的不同版本、用 fallback 做 CDN 容灾。

### 2.3 怎么用

在页面里、在任何 `<script type="module">` 之前放一份映射即可：

```html
<script type="importmap">
{
  "imports": {
    "lodash": "https://esm.dancf.com/npm:lodash-es@4.17.21/lodash.js"
  }
}
</script>
<script type="module">
  import _ from 'lodash';
  _.clone({});
</script>
```

之后模块里的 `import _ from 'lodash'` 会由 Import Map 解析到对应 URL。

## 三、WPM 是什么、解决什么、怎么用

### 3.1 是什么

WPM（Web Package Manager）是自研的 **Import Map 生成工具**：

- **WPM1**：初版，能根据依赖生成 import map，能力有限。
- **WPM2**：按真实 node_modules 版本生成、支持子路径（如 `lit/decorators.js`）、保证开发/生产版本一致。
- **WPM3**：在 WPM2 上增加对 **Monorepo Workspace 包** 的支持，用 Git 提交 hash 当版本，能和 npm 包的 import map 合并。

### 3.2 解决什么问题

- 手写、手维护一大块 `imports` / `scopes` 容易错，且和 package.json / lockfile 不同步。
- WPM 从「当前项目真实装好的依赖」出发，自动解析依赖树，生成与本地一致的 Import Map，避免「本地一个版本、线上/Import Map 里另一个版本」。

### 3.3 怎么用（概念上）

- 在项目里配置要参与生成 import map 的包（或直接用依赖声明）。
- 在构建/发布流程里跑 WPM（如 CI 或本地脚本）。
- 把 WPM 输出的 JSON 注入到 HTML 的 `<script type="importmap">` 里（或由构建脚本插入），页面里的 ESM 就能通过这份 map 正确解析裸模块名。

## 四、和「打包」的关系：开发 vs 构建 vs 运行时

### 4.1 容易产生的疑问

- 开发阶段：`import xx from 'xx'` 是从 node_modules 引入的。
- 打包之后：HTML 里插入了 Import Map，**还需要打包该包吗？** 如果已经打包了，**如何知道引用该包？**

### 4.2 关键：构建时把这些包设为「外部化」（external）

使用 Import Map 时，会把这些依赖**从打包里排除**，而不是打进 bundle：

```javascript
// 例如 Vite / Rollup
build: {
  rollupOptions: {
    external: ['vue', 'react', 'react-dom'],  // 与 Import Map 中的包一致
  }
}
```

**external 的含义**：这些包**不参与打包**，产物的 JS 里会**保留**原来的 `import Vue from 'vue'` 语句，不会把 Vue 的源码打进 bundle。

### 4.3 还需要打包该包吗？

**不需要。** 出现在 Import Map 里的包：

- **构建时**：不打包（external）。
- **运行时**：由浏览器根据 Import Map 从 CDN 加载。

### 4.4 如何知道引用该包？

不是「包也被打包进去了」，而是「包被排除在 bundle 外，产物里还留着 `import vue from 'vue'`」：

1. HTML 里先有 `<script type="importmap">`，例如 `"vue"` → `"https://esm.dancf.com/npm:vue@2.7.13/..."`。
2. 浏览器加载你的 `app.xxx.js`。
3. 执行到 `import Vue from 'vue'` 时，浏览器看到裸说明符 `'vue'`，查 Import Map，得到 CDN URL，去拉取 Vue 的 ESM。

所以：**构建只负责保留这句 import；运行时由 Import Map 告诉浏览器去哪个 URL 拿。**

### 4.5 和传统打包的对比

| 阶段   | 传统打包（没有 Import Map）     | 使用 Import Map + external           |
|--------|----------------------------------|--------------------------------------|
| 构建时 | 把 vue 等依赖都打进 bundle       | 把 vue 等设为 external，**不打进** bundle |
| 产物   | 没有 `import 'vue'`，只有打包后的代码 | **仍有** `import Vue from 'vue'`     |
| 运行时 | 只请求你的 bundle，依赖已在里面  | 先请求你的 bundle，再按 Import Map 请求 CDN 上的 vue |

**总结三句**：

1. **开发时**：`import xx from 'xx'` 从 node_modules 解析，和现在一样。
2. **构建时**：这些包被 external，**不打包**，产物里**保留** `import xx from 'xx'`。
3. **运行时**：浏览器看到 `import xx from 'xx'`，查 HTML 里的 Import Map，得到 CDN URL，再去拉该包。

## 五、这么做的好处

### 5.1 缓存更好、更新成本更低

- **传统**：依赖升级 → 引用它的模块内容变 → 整条链 hash 变 → 整包缓存失效。
- **Import Map**：依赖在 CDN，URL 带版本；升级时只改 Import Map 里那一行，业务 bundle 不变 → 业务文件缓存仍有效，用户只多下一次新版本依赖。

### 5.2 依赖升级可「只改配置、不重打整包」

- **传统**：改一个依赖 → 必须重新构建整个项目 → 部署一整套新资源。
- **Import Map**：改依赖版本 = 只更新 import map（或生成 import map 的配置），业务 bundle 可不变，部署和回滚都更快。

### 5.3 构建更快、产物体积更小

- 大依赖不参与打包，构建时间明显缩短（文档中有项目从约 7 分钟降到约 3 分钟）。
- 你的 bundle 里少了一大块依赖代码，产物体积会减小（文档有约 20% 的案例）。

### 5.4 跨应用共享同一份依赖

多个应用都用同一 CDN 的 Vue/React（同一 URL、同一版本），用户访问过 A 再访问 B 时可直接复用缓存，无需再下一次 Vue/React。对微前端、多产品线共用框架特别有用。

### 5.5 部署与标准化

- **SaaS / 多租户**：同一套前端资源 + 不同 import map，即可在不同环境用不同依赖版本或不同 CDN。
- **包规范**：依赖以 ESM 形式从 CDN 提供，推动使用 `exports`、ESM、更清晰的依赖声明。

### 5.6 文档中的收益小结

| 维度     | 收益概要 |
|----------|----------|
| **性能** | 构建更快、产物体积更小、缓存命中率更高 |
| **发布** | 依赖更新只改 Import Map，减少全量构建和全量失效 |
| **体验** | 大依赖不参与打包，多应用可共享 CDN 依赖 |
| **架构** | 跨应用共享、版本隔离（Scopes）、微前端/动态加载更自然 |

## 六、关系小结

|        | 是什么                 | 解决什么问题                         |
|--------|------------------------|--------------------------------------|
| **Import Map** | 浏览器里的「包名 → URL」映射表 | 在浏览器里用裸模块名、做缓存和版本隔离 |
| **WPM**        | 生成这份映射表的工具     | 自动、准确、和 node_modules 一致，支持 npm + workspace |

**使用顺序**：用 WPM 生成 Import Map → 把生成的 JSON 放到页面 → 页面里就可以 `import 'xxx'` 并由 CDN 解析。

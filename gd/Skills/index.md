# Skills 说明

## 什么是 Skill？

**Skill**（技能）是 Cursor 里的一种「教 AI 怎么做某类事」的配置。它是一份 Markdown 文档（通常放在 `SKILL.md` 里），用自然语言和示例告诉 AI：

- **做什么**：这类任务的目标和范围  
- **何时用**：在什么场景、什么用户意图下应该启用  
- **怎么做**：步骤、格式、约定和示例  

AI 会根据对话内容和 Skill 的 **description** 自动判断是否启用某个 Skill，从而按你定义的流程、格式或规范来执行任务。

### 典型用途

| 用途       | 说明                                               |
| ---------- | -------------------------------------------------- |
| 代码审查   | 按团队规范做 PR/代码审查，统一反馈格式             |
| 提交信息   | 根据 diff 生成符合约定的 commit message            |
| 文档/报告  | 按固定模板生成分析、报告或说明                     |
| 领域工作流 | 查库表、跑脚本、填表等固定流程的标准化             |
| 项目约定   | 命名、目录结构、技术选型等只在项目内有效的知识     |

---

## Skill 的存放位置

- **个人**：`~/.cursor/skills/<skill-name>/` — 所有项目共用
- **项目**：`.cursor/skills/<skill-name>/` — 当前仓库，可随代码共享

注意：不要使用 `~/.cursor/skills-cursor/`，该目录为 Cursor 内置技能保留。

---

## 基本结构

一个 Skill 对应一个目录，其中**必须**包含 `SKILL.md`：

```text
skill-name/
├── SKILL.md              # 必需：主要说明与步骤
├── reference.md          # 可选：详细参考
├── examples.md           # 可选：更多示例
└── scripts/              # 可选：可执行脚本
```

### SKILL.md 的必备部分

1. **YAML 头信息（frontmatter）**
   - `name`：技能唯一标识，建议小写+连字符，如 `code-review`
   - `description`：简短说明「做什么 + 何时用」，供 AI 做匹配

2. **正文**
   - 步骤说明、检查项、输出格式、示例等，用 Markdown 写清楚即可。

示例头信息：

```yaml
---
name: code-review
description: 按团队规范做代码审查，输出分级反馈。在用户要求审查 PR、看代码改动或提到 code review 时使用。
---
```

## 如何写好 Skill

### 1. 把 description 写清楚

- 用**第三人称**写（会被注入系统提示），例如：「处理 Excel 并生成报表」而不是「我可以帮你处理 Excel」。
- 同时写清 **WHAT**（能做什么）和 **WHEN**（什么情况下用），并带上触发关键词（如：PDF、commit message、code review）。

### 2. 保持精简

- AI 本身能力已经较强，只补充它**不知道**的：项目约定、团队习惯、固定格式。
- 主文件 `SKILL.md` 建议控制在约 500 行内，细节放到 `reference.md`、`examples.md` 等，需要时再让 AI 去读。

### 3. 用「渐进式展开」

- 核心步骤、必读规则放在 `SKILL.md`。
- 详细 API、长示例、历史约定放在引用文档里，并在 SKILL 里用一两句话说明「详见 xxx」。

### 4. 善用模板与示例

- 需要固定输出格式时，在 SKILL 里直接给 Markdown/代码模板。
- 需要统一风格时，给 1～2 个完整「输入 → 输出」示例，比大段抽象描述更有效。

### 5. 需要严格一致时用脚本

- 若某步必须和某脚本/命令一致（如校验、迁移），在 Skill 里写明「调用 `scripts/xxx`」，并说明是「执行」还是「仅作参考」。脚本比让 AI 临时写代码更稳定、省 token。

## 常见模式简表

| 模式         | 适用场景                       | 做法                                             |
| ------------ | ------------------------------ | ------------------------------------------------ |
| **模板**     | 报告、总结、提交信息等固定格式 | 在 SKILL 中给出 Markdown/文本模板                |
| **示例**     | 格式、语气、粒度要求           | 给出 1～2 个完整示例（输入 + 期望输出）          |
| **工作流**   | 多步骤、有顺序的任务           | 用步骤列表 + 可选检查清单                        |
| **条件分支** | 不同情况不同流程               | 用「若 A → 做 X；若 B → 做 Y」写清分支           |
| **校验循环** | 质量要求高、需自检             | 步骤里明确「先执行脚本/检查，失败则修正再重跑」  |

## 本目录（gd/Skills）说明

本目录用于存放与 **Skills** 相关的说明与索引（即本文档）。  
若要为 Cursor 编写或管理 Skill，可参考：

- Cursor 官方/社区关于 Agent Skills 的说明  
- 项目或团队内约定的 Skill 命名、存放位置（个人 vs 项目）  
- 上文中的结构、description 写法与常见模式  

如需创建新 Skill，可在 Cursor 中直接说「帮我创建一个 Skill，用来……」并说明用途和触发场景，按上述要点即可生成并迭代 `SKILL.md` 与配套文件。
